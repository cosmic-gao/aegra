Aegra uses OpenTelemetry for all observability. This means you can send traces to multiple backends simultaneously without changing your code — no vendor lock-in.

## Supported backends

Out of the box, Aegra supports:

- **Langfuse** — Production-grade LLM observability
- **Arize Phoenix** — Local debugging and evaluation
- **Generic OTLP** — Any compatible backend (Jaeger, Honeycomb, Datadog, etc.)

You can enable one, multiple, or all of these at the same time.

## Configuration

Tracing is configured entirely through environment variables in your `.env` file.

### Enable tracing

Set the `OTEL_TARGETS` variable to a comma-separated list of backends:

```bash
# Enable Langfuse and Phoenix simultaneously
OTEL_TARGETS="LANGFUSE,PHOENIX"

# Enable only generic OTLP
OTEL_TARGETS="GENERIC"

# Disable all tracing (default)
OTEL_TARGETS=""
```

For debugging, you can also log traces to the console:

```bash
OTEL_CONSOLE_EXPORT=true
```

### Fan-out to multiple backends

Send traces to multiple backends at once by listing them in `OTEL_TARGETS`:

```bash
OTEL_TARGETS="LANGFUSE,PHOENIX"
LANGFUSE_BASE_URL=https://cloud.langfuse.com
LANGFUSE_PUBLIC_KEY=pk-lf-...
LANGFUSE_SECRET_KEY=sk-lf-...
PHOENIX_COLLECTOR_ENDPOINT=http://127.0.0.1:6006/v1/traces
```

Each backend receives the same trace data through separate `BatchSpanProcessor` instances.

### Redacting LLM inputs and outputs (PII)

By default, auto-instrumentation exports the full LLM payload — prompts, completions, end-user messages, and tool-call arguments — to every configured backend. In multi-tenant or regulated deployments this is the largest PII/compliance surface. Two switches redact it at the source, before any span leaves the process:

```bash
# Replace LLM prompts / input messages with a redacted marker
OTEL_HIDE_LLM_INPUTS=true
# Replace LLM completions / output messages with a redacted marker
OTEL_HIDE_LLM_OUTPUTS=true
```

These map to OpenInference's `TraceConfig(hide_inputs=…)` / `hide_outputs=…`, which redact the input/output values, prompt/completion messages, and tool arguments while leaving span structure, timing, token counts, and metadata intact.

Both default to `false` to preserve existing behavior. A `false` switch does **not** force redaction off — it defers to OpenInference's native `OPENINFERENCE_HIDE_INPUTS` / `OPENINFERENCE_HIDE_OUTPUTS` environment variables, so any redaction you already rely on keeps working. Setting a switch to `true` always redacts, regardless of those env vars.

<Note>
**User and session identifiers are always attached.** Even with redaction on, every span still carries `user.id` / `langfuse.user.id` and `session.id` / `langfuse.session.id` (set in `observability/span_enrichment.py`). This is a deliberate identity annotation — Langfuse user/session analytics depend on it. These are opaque identifiers (the authenticated identity and the thread id), not message content.
</Note>

### Sampling

By default every trace is exported (the SDK default is `parentbased_always_on`). At high request volume you can export only a fraction of traces to cut client cost and backend ingestion, using the standard OpenTelemetry sampler variables:

```bash
# Export ~10% of runs; respect an upstream sampling decision when one is present
OTEL_TRACES_SAMPLER=parentbased_traceidratio
OTEL_TRACES_SAMPLER_ARG=0.1
```

Supported samplers: `always_on`, `always_off`, `traceidratio`, `parentbased_always_on`, `parentbased_always_off`, `parentbased_traceidratio`. `OTEL_TRACES_SAMPLER_ARG` is the sampling ratio (`0.0`–`1.0`) and applies only to the ratio-based samplers. Leaving `OTEL_TRACES_SAMPLER` empty keeps the default (export everything); an unknown name or an out-of-range ratio is ignored with a warning and also falls back to the default.

Because a run's trace id is derived deterministically from its `run_id` (see [Trace identity](#trace-identity)), ratio sampling is **consistent per run**: every span in a run shares one trace id and therefore one sampling decision, so a run's trace is never partially sampled — you never get orphaned child spans. When trace context crosses the Redis worker boundary, prefer a `parentbased_*` sampler so the whole distributed trace inherits the root's decision.

## How it works

Aegra uses a pure OpenTelemetry approach:

1. **Auto-instrumentation** captures the whole request path — LangGraph steps (`openinference-instrumentation-langchain`), HTTP server spans for every route including your custom routes (`opentelemetry-instrumentation-fastapi`), and outbound HTTP client calls with W3C `traceparent` propagation to downstream services such as sandboxes and tool servers (`opentelemetry-instrumentation-httpx`)
2. **Singleton provider** is initialized once during application startup
3. **Fan-out** sends the same trace data to all configured exporters via `BatchSpanProcessor`

This keeps overhead low and maintains compatibility with the entire OpenTelemetry ecosystem. All auto-instrumentation is enabled only when at least one target (or console export) is configured.

### Trace identity

A run's trace id **equals its `run_id`**: the 128-bit OTEL trace id is the run's UUID, so the 32-hex trace id in Langfuse is simply the `run_id` with its dashes removed. This mirrors LangGraph Platform / LangSmith, where a trace is identified by its root run.

Because the mapping is deterministic, any service holding the `run_id` (returned by the run API) can compute the trace id with no lookup — the recommended way to attach scores, evaluations, or user feedback to a run's trace after it finishes:

```python
trace_id = run_id.replace("-", "")   # 32-hex OTEL trace id for the run
```

A non-UUID run id (for example a client-supplied A2A task id) is hashed deterministically into a 128-bit trace id instead.

### Execution surfaces

Every way a graph can run is enriched identically — the run's root span and all of its child observations carry `langfuse.user.id`, `langfuse.session.id`, and `langfuse.trace.name` (the graph id), so filtering by assistant in Langfuse surfaces every invocation regardless of entry point:

- **Runs API** — `POST /threads/{thread_id}/runs`, `/runs/stream`, `/runs/wait`, plus cron- and delayed-run–triggered runs
- **A2A** — `/a2a/{assistant_id}`; traces are tagged `source=a2a`
- **MCP** — `/mcp` (one tool per graph); traces are tagged `source=mcp`

## Run metadata

Each `POST /threads/{thread_id}/runs` (and `/runs/stream`, `/runs/wait`) request accepts an optional top-level `metadata` field whose key/value pairs are propagated onto the run's root OTEL span. This is the recommended channel for filterable trace attributes that aren't part of the LangGraph payload — tenant id, feature flag, environment tag, sub-agent type, and similar.

```json
{
  "assistant_id": "agent",
  "input": {"question": "..."},
  "metadata": {
    "tenant": "acme",
    "feature_flag": true,
    "subagent": "matter_legal"
  }
}
```

Each entry reaches the root span as `langfuse.trace.metadata.<key>`, which Langfuse exposes as a queryable trace property (rather than burying it under per-observation metadata). On Phoenix and other OTLP targets the same attribute is set verbatim; native first-class metadata aliasing is a planned follow-up.

The value is also persisted to the runs table (`execution_params.run_metadata` JSONB column) so it survives worker restart and is available for post-hoc analysis.

### Constraints

The request validator rejects payloads that would either be silently dropped downstream or balloon span size past collector limits. A violating payload returns `422` with a single message identifying the offending key:

| Constraint | Limit |
|------------|-------|
| Maximum number of keys | 32 |
| Key character set | `[A-Za-z0-9_-]` (no dots, no whitespace, no non-ASCII) |
| Key length | 1–64 characters |
| Value type | `str`, `int`, `float`, or `bool` (no nested dicts, lists, or `null`) |
| String value length | ≤ 512 characters |

The dot exclusion is deliberate: keys are stored under the `langfuse.trace.metadata.` prefix, and allowing dots would let a caller land bare attributes (e.g. `langfuse.user.id`) next to the system ones.

### System-key collisions

Aegra injects a small number of runtime keys into the same metadata stream so they're filterable alongside user-supplied attributes:

- `run_id`, `thread_id`, `graph_id` — always present
- `original_request_id` — present on the worker path when an HTTP correlation-id was supplied

If the request `metadata` contains a key already populated by the runtime, the system value wins, the user value is dropped, and a warning is logged from `aegra_api.observability.span_enrichment`. This makes the OTEL view a reliable join key with logs and the runs table; user audit fidelity is preserved by keeping the original payload in `execution_params.run_metadata`.

## Key environment variables

The main variables you'll need:

| Variable | Description |
|----------|-------------|
| `OTEL_TARGETS` | Comma-separated list of backends: `LANGFUSE`, `PHOENIX`, `GENERIC` |
| `OTEL_SERVICE_NAME` | Service name for traces (default: `aegra-backend`) |
| `OTEL_CONSOLE_EXPORT` | Log traces to console (`true`/`false`) |
| `OTEL_HIDE_LLM_INPUTS` | Redact LLM prompts / input messages before export (`true`/`false`, default `false`) |
| `OTEL_HIDE_LLM_OUTPUTS` | Redact LLM completions / output messages before export (`true`/`false`, default `false`) |
| `OTEL_TRACES_SAMPLER` | Trace sampler: `always_on`, `traceidratio`, `parentbased_traceidratio`, … (empty = export all) |
| `OTEL_TRACES_SAMPLER_ARG` | Sampling ratio `0.0`–`1.0` for ratio-based samplers (default `1.0`) |

Each provider has its own set of variables (endpoints, API keys). See the [environment variables reference](/reference/environment-variables) for the full list including all Langfuse, Phoenix, and generic OTLP variables.

## Prometheus metrics

For infrastructure-level monitoring (request rates, latency, error rates), Aegra supports an optional Prometheus metrics endpoint alongside OpenTelemetry tracing.

```bash
ENABLE_PROMETHEUS_METRICS=true
```

This exposes a `/metrics` endpoint with standard HTTP request metrics in Prometheus exposition format. Scrape it with any Prometheus-compatible collector and visualize with Grafana.

The `/metrics` endpoint is **not** protected by Aegra's authentication middleware. This is intentional — Prometheus scrapers typically do not support application-level auth. If you need to restrict access, use network-level controls (firewall rules, internal load-balancer listeners, etc.).

See the [environment variables reference](/reference/environment-variables#prometheus-metrics) for details.
